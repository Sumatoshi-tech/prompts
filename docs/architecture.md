# Architecture

> Internal design documentation for contributors. For user-facing docs, see [Commands](commands.md), [Configuration](configuration.md), [Workflows](workflows.md), and [Templates](templates.md).

## Project Structure

```
promptkit/
  cmd/promptkit/main.go                # Entry point
  internal/
    cli/                               # Cobra commands
      init.go                          #   promptkit init
      update.go                        #   promptkit update (core pipeline)
      diff.go                          #   promptkit diff (+ --upstream)
      status.go                        #   promptkit status
      doctor.go                        #   promptkit doctor
      clean.go                         #   promptkit clean
      template.go                      #   promptkit template (list/add/extract/render/vars)
      config.go                        #   promptkit config explain
      root.go                          #   Root command and global flags
    config/                            # .promptkit.yaml schema, load, save, validate, migrate
      ecosystem.go                     #   Ecosystem module registry
      eco_golang.go                    #   Go ecosystem module
      eco_rust.go                      #   Rust ecosystem module
      eco_zig.go                       #   Zig ecosystem module
    scaffold/                          # Template rendering, diffing, file I/O, locking, overrides
    adapters/                          # Agent-specific file placement (6 agents)
    prompt/                            # Interactive terminal prompts
  templates/
    golang/                            # Go ecosystem templates
    rust/                              # Rust ecosystem templates
    zig/                               # Zig ecosystem templates
  templates.go                         # embed.FS declaration
```

## Ecosystem Module System

Each ecosystem is a self-contained module registered via `config.RegisterEcosystem()`. A module provides:

- **DefaultCmdPath** -- default binary entry point (e.g. `./cmd/name` for Go, `src/main.rs` for Rust)
- **ApplyDefaults** -- set ecosystem-specific default values
- **Validate** -- ecosystem-specific validation rules
- **RegisterFlags / ApplyFlags** -- CLI flag registration and application
- **RunPrompts** -- interactive prompts for ecosystem-specific fields
- **CommentedFields** -- YAML output formatting
- **FieldEntries** -- field-to-file mapping metadata

Adding a new ecosystem means creating a single `eco_<name>.go` file with an `init()` function.

## Agent Adapter System

The adapter layer transforms shared instruction templates into agent-specific file formats. Each agent has a placement strategy:

- **Claude Code**: `.claude/commands/<skill>.md`
- **Codex CLI**: reads `.agents/skills/` and `AGENTS.md` natively
- **GitHub Copilot**: `.github/copilot-instructions.md` (concatenated)
- **Cursor**: `.cursor/rules/agents.mdc`
- **Gemini CLI**: `GEMINI.md` + `.gemini/commands/<skill>.toml`
- **Windsurf**: `.windsurfrules` + `.windsurf/workflows/<skill>.md`

Skills are workflow-aware only in text: `instr-implement.md` and `AGENTS.md` reference either FRD or journey artifacts. Base skills are `/implement`, `/roadmap`, and `/perf`. `instr-frd.md` / `instr-journey.md` remain as files under `.agents/instructions/` (one per project), not as Agent Skills.

## Safety Model

- **Diff-then-approve**: Every destructive operation shows a unified diff first
- **Atomic writes**: Files are written to temp, then renamed -- no partial writes
- **Timestamped backups**: `.promptkit/backups/<timestamp>/` created before every update
- **Rollback on failure**: If apply fails mid-way, backed-up files are restored automatically
- **Concurrent execution lock**: PID-based `.promptkit/lock` prevents two instances from running simultaneously
- **Stale lock detection**: If a previous run crashed, the stale lock is reclaimed automatically
- **Manifest tracking**: `generated_files` in config tracks every file promptkit manages
- **Checksum tracking**: Detects manually modified files and warns in verbose mode
- **Provenance comments**: Every generated file includes a "Generated by promptkit" header
- **Merge conflict detection**: Config loading fails with actionable message if conflict markers found

## Team Standardization

promptkit supports team-wide standards through scripted initialization:

```bash
#!/bin/bash
# team-bootstrap.sh -- apply team standards to a project

PROJECT_DIR="${1:-.}"
TEAM_CONFIG="https://raw.githubusercontent.com/org/standards/.promptkit.yaml"

# Copy team config
curl -sL "$TEAM_CONFIG" > "$PROJECT_DIR/.promptkit.yaml"

# Apply with project-specific overrides
cd "$PROJECT_DIR"
promptkit init --force --non-interactive \
  --name "$(basename "$PWD")" \
  --module "github.com/org/$(basename "$PWD")"
```

### Drift Detection in CI

```yaml
# .github/workflows/lint.yml
- name: Check promptkit drift
  run: promptkit status
```

`promptkit status` exits with code 1 if any generated files are out of date, catching configuration drift in CI before it reaches production.

### Comparing Configurations

```bash
# Compare your project against the team reference
promptkit diff --upstream /path/to/team-standard-project
```
